1. Introduction

1.1 Project Statement

The objective of this project is to design, implement, and functionally verify a high-performance, 4-wide Superscalar Out-of-Order (OoO) RISC-V processor in synthesizable Verilog HDL. The processor targets the RV32I base integer instruction set with the M-extension (MUL, MULH, MULHU, MULHSU), and can fetch, decode, rename, issue, execute, and commit up to four instructions per clock cycle.

Unlike in-order pipelined processors that execute strictly in program order, this OoO core dynamically schedules instructions based on operand availability, maximizing functional unit utilization and throughput. The project spans the full hardware design lifecycle: RTL design, functional simulation using cocotb, and development of an automated Python-based verification and visualization suite. A classic 5-stage in-order pipelined RISC-V processor was also designed as a baseline reference architecture.

1.2 Background Survey

Modern high-performance processors universally employ out-of-order execution to extract maximum performance. The foundational concepts trace back to Robert Tomasulo's algorithm (1967), developed for the IBM System/360 Model 91. His key innovations — register renaming to eliminate false dependencies (WAW/WAR hazards) and reservation stations for dynamic scheduling — remain the architectural pillars of every OoO engine today.

The RISC-V open ISA has become the preferred vehicle for processor research due to its clean, modular design. While open-source OoO RISC-V cores exist (e.g., BOOM from UC Berkeley), most are in Chisel/SystemVerilog and target SoC integration. This project implements a complete OoO engine in pure Verilog, emphasizing pedagogical clarity, modular design, and end-to-end verification. Key references include Patterson & Hennessy's textbooks, the RISC-V ISA Manual (v2.2), and UC Berkeley's BOOM architecture.

2. Methodology

2.1 Techniques & Approach

Phase 1 — Baseline 5-Stage Pipeline: A classic in-order RV32I processor was designed with full data hazard resolution (forwarding MUX network), control hazard handling (branch-in-ID), and load-use hazard detection. This 17-module baseline serves as the performance reference.

Phase 2 — 4-Wide Superscalar OoO Core: Building on Phase 1, the full OoO core was designed with:

  • 4-Wide Superscalar Frontend: Fetches 4 instructions/cycle with a Pre-decode & Branch Prediction Layer for early branch target identification and valid-bit masking.
  • Register Renaming (RAT + Free List + PRF): A 32-entry Register Alias Table maps architectural registers to 128 physical registers, completely eliminating WAW and WAR false dependencies.
  • Dynamic Scheduling (16-entry Issue Queue): Dispatched instructions wait until operands are ready. A CDB wakeup network monitors 4 broadcast tags/cycle. Dedicated MUL-priority selection on execution port 3.
  • Speculative Execution & Recovery: A 32-entry Reorder Buffer (ROB) supports speculative fetch past branches and performs sequential walkback on mispredictions to restore precise architectural state.
  • Load-Store Queue (8-entry LSQ): Manages memory ordering with store-to-load forwarding and prevents loads from bypassing stores with unresolved addresses.

3.2 Performance Analysis

Peak IPC Achieved (Max ILP Test): 3.20 / 4.00 theoretical maximum (80.0% efficiency)
Worst-Case IPC (RAW Chain): 0.88 / 1.00 (serial dependency bound)
Fibonacci IPC: 0.95 (dual-operand dependent chain)

The peak IPC of 3.2 demonstrates that the 4-wide commit path and execution engine operate at 80% of the theoretical maximum on fully independent instructions. The gap from 4.0 is attributed to pipeline fill/drain latency and CDB structural hazards. The RAW chain test correctly serializes to near-1.0 IPC, confirming proper dependency enforcement through the Issue Queue wakeup logic. The branch misprediction test verifies correct speculative state recovery via ROB walkback, with the final architectural state (x2 = 6) matching the expected loop sum.

4. Challenges & Learnings

4.1 Obstacles Faced

  1. Branch Misprediction Recovery (ROB Walkback): Implementing correct speculative state recovery was the most complex challenge. When a misprediction is detected, the ROB must sequentially walk back from tail to head, restoring each RAT entry. Initial implementations caused deadlocks between ROB commit logic and the squash FSM.
  2. Load-Store Queue Memory Ordering: Ensuring loads could not bypass older stores with unresolved addresses was a subtle correctness issue. An early implementation allowed speculative loads, leading to incorrect reads. The fix required scanning the entire LSQ for older unresolved stores before permitting any load to issue.
  3. CDB Broadcast Contention: With 3 ALU ports, 1 multiplier port, and 1 LSQ port producing results simultaneously, CDB contention required careful arbitration. The solution involved dedicated CDB channels and priority logic.
  4. Free List Exhaustion Stalls: With 128 physical registers serving a 4-wide rename stage, the Free List could become exhausted during long programs. Tuning the ROB commit width to 4-wide (matching rename width) was essential to keep the Free List replenished.
  5. Synthesis Latch Elimination: Initial RTL contained unintentional latches from incomplete sensitivity lists. These were systematically eliminated by refactoring all combinational blocks with proper default assignments.
  6. Windows/WSL Path Interoperability: Running cocotb through WSL with Windows-resident source files caused path-resolution failures. The verification wrapper had to clone RTL files to a sanitized staging directory.

4.2 Solutions & Insights

  • Modular FSM Design: The ROB walkback was isolated into a dedicated squashing state variable, decoupling recovery logic from normal commit and preventing deadlocks.
  • Store Forwarding with Safety Checks: The LSQ implements a two-phase check — (1) verify no older store has an unresolved address, (2) scan for address-matching stores to forward data — achieving correctness without unnecessary serialization.
  • Dedicated MUL Port: Assigning a 3-stage pipelined multiplier to execution port 3 with MUL-priority selection eliminated structural hazards between ALU ops and multi-cycle multiplies.
  • Automated Verification: The Python/cocotb wrapper automates the entire flow — writing program.mem, invoking simulation, capturing commits, computing IPC, and verifying registers against golden values — reducing debug time by an order of magnitude.

5. Conclusion & Applications

5.1 Final Takeaways

This project successfully demonstrates the complete design lifecycle of a 4-wide superscalar OoO RISC-V processor. Key accomplishments:

  • 26 Verilog modules for the OoO pipeline + 17 modules for the baseline 5-stage pipeline — all from scratch in portable, synthesizable Verilog.
  • A peak IPC of 3.2 out of a theoretical 4.0 (80% efficiency) on independent instruction streams.
  • 100% functional correctness across all 8 benchmarks covering data hazards (RAW, WAW, WAR), control hazards (branch misprediction, JAL/JALR), memory operations (store-to-load forwarding), and M-extension multiplier verification.
  • A robust Python/cocotb verification suite with an offline HTML expo dashboard for automated regression testing and interactive performance visualization.

5.2 Real-World Applications

  • Educational Reference: Clean, modular codebase for graduate computer architecture courses — hands-on Tomasulo's algorithm, register renaming, ROB-based commit, and LSQ design.
  • Embedded HPC: Foundation for high-performance RISC-V embedded processors in robotics, autonomous systems, and edge AI inference (with cache/TLB augmentation).
  • DSP & Cryptographic Acceleration: The M-extension multiplier enables compute-intensive workloads like signal processing and cryptographic hashing.
  • Open-Source Silicon: Fully open-source design suitable for RISC-V ecosystem contributions and open silicon tape-outs.

6. Future Work & Enhancements

  • L1 Cache Integration: Split I-cache / D-cache hierarchy to replace register-based memories for practical program sizes.
  • Advanced Branch Prediction: 2-bit saturating counter predictor with BTB and Return Address Stack to reduce misprediction penalties.
  • Wider Issue & Execute: Scaling to 6/8-wide issue with heterogeneous functional units (FPU, SIMD).
  • Precise Exception Support: RISC-V privilege architecture (M/S/U modes, CSRs, trap vectors) for OS boot capability.
  • ASIC Tape-Out: Porting to OpenLane with SkyWater 130nm PDK for gate-level area/power/frequency metrics.
  • Multi-Core Scaling: Multiple OoO cores with shared L2 cache and coherence protocol (MESI/MOESI).
